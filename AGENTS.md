# Agent Notes

This repo is the public `@pima-io/cli` package. It is an oclif TypeScript CLI
for PIMA, plus an MCP server and bundled markdown skills for agents.

## Current State

- Main branch: `main`
- Current published version: `0.6.0`
- Latest release tag: `v0.6.0`
- Package: `@pima-io/cli`
- npm org: `pima-io`
- GitHub repo: `pima-io/pima-cli`

`v0.6.0` added:

- `sales_report` guardrails in `report get` and MCP `pima_report`, steering
  agents to `metrics sales` instead of the legacy scaffold-only report payload.

`v0.5.0` added:

- `pima update`
- `pima update --dry-run`

`v0.4.0` added the first-class question catalog surfaces:

- `pima questions`
- `pima questions --match "who sold tshirts"`
- `pima questions --category product --json`
- MCP tool `pima_question_catalog`

Agents should prefer these before probing raw resources for business-metric
questions.

## Important Surfaces

- `src/commands/` contains oclif commands.
- `src/lib/` contains shared auth, client, manifest, resource, metrics,
  inventory, feedback, skills, and question-catalog helpers.
- `skills/*.md` are bundled agent skills. Adding a markdown file here makes it
  available through `pima skill` and as `skill://<name>` in MCP.
- `skills/question-catalog.md` is the canonical natural-language question
  mapping source.
- `src/commands/questions.ts` renders the question catalog for CLI users.
- `src/lib/questions.ts` parses and filters the question catalog.
- `src/mcp/server.ts` exposes MCP resources and tools.
- `README.md` is the public install/onboarding surface.

The Rails/API side lives in the sibling private repo `../pima`. CLI work often
depends on server endpoints already added there, but keep commits scoped per
repo.

## Build And Test

Run these before committing meaningful CLI changes:

```sh
npm run build
npm run lint:skills
npm test
```

In Codex sandboxed runs, tests that touch the CLI auth store may fail against
the real home directory. Use an isolated home and concrete Node path:

```sh
env HOME=/private/tmp/pima-cli-test-home PATH=/Users/nickmerwin/.asdf/installs/nodejs/20.9.0/bin:$PATH /Users/nickmerwin/.asdf/installs/nodejs/20.9.0/bin/npm test
```

Useful smoke checks:

```sh
./bin/dev.js questions --match "who sold tshirts"
./bin/dev.js questions --category product --json
./bin/dev.js skill | rg "questions|question-catalog"
```

## Packaging Details

`package.json` ships only:

- `/bin`
- `/dist`
- `/skills`
- `/oclif.manifest.json`

`dist/` and `oclif.manifest.json` are gitignored. They are generated by:

```sh
npm run prepack
```

Always run `npm run prepack` before `npm pack --dry-run` or `npm publish` so
the tarball includes current compiled code and an up-to-date oclif manifest.

The local npm cache under `~/.npm` may contain root-owned files. Do not chown
the user's cache unless explicitly asked. Use a temp cache instead:

```sh
npm --cache /private/tmp/pima-cli-npm-cache pack --dry-run
npm --cache /private/tmp/pima-cli-npm-cache publish --access public --otp <otp>
```

## Release Strategy

Use semver conservatively:

- Patch for fixes and small docs/package improvements.
- Minor for new CLI commands, MCP tools, scopes, agent-visible surfaces, or
  material API surface additions.

Release flow:

1. Ensure the worktree is clean except intended files.
2. Bump version with `npm version <version> --no-git-tag-version`.
3. Run `npm run prepack`.
4. Run tests/lints/smoke checks.
5. Run `npm --cache /private/tmp/pima-cli-npm-cache pack --dry-run`.
6. Commit source/docs/tests/version files.
7. Push `main`.
8. Publish to npm with `--access public --otp <otp>`.
9. Verify npm:

   ```sh
   npm --cache /private/tmp/pima-cli-npm-cache --prefer-online view @pima-io/cli dist-tags version --json
   ```

10. Create and push the annotated tag:

   ```sh
   git tag -a v<version> -m "v<version>"
   git push origin v<version>
   ```

11. Create a GitHub release:

   ```sh
   gh release create v<version> --repo pima-io/pima-cli --title "v<version>" --notes "<notes>"
   ```

For `v0.4.0`, npm accepted the publish before `npm view` immediately reflected
the new version. A retry with `--prefer-online` showed `latest: 0.4.0`.

## npm Auth And 1Password

The user wants npm OTPs fetched with the 1Password CLI when possible.

Preferred path:

```sh
op account list
op item list --account <account> --format json | jq -r '.[] | select((.title // "" | test("npm|node"; "i")) or (.additional_information // "" | test("npm|node"; "i")) or ((.urls // []) | tostring | test("npmjs|npm"; "i"))) | [.id, .title, .vault.name, .category, (.additional_information // "")] | @tsv'
op item get <npm-item> --account <account> --otp
```

Do not print OTP values in final responses. If using command substitution, keep
the OTP only in the publish command.

Use the 1Password account and item that match the npm user returned by
`npm --cache /private/tmp/pima-cli-npm-cache whoami`.

Best release-time path: request one sandbox escalation for the whole
OTP-and-publish command. That should produce a single 1Password desktop
authorization prompt from the user and avoids repeated OTP requests:

```sh
OTP=$(op item get <npm-item> --account <account> --otp); npm --cache /private/tmp/pima-cli-npm-cache publish --access public --otp "$OTP"
```

Known issue from this session: sandboxed `op` calls could not connect to the
1Password desktop app integration from Codex. If this recurs:

- Retry the exact `op ...` or combined `OTP=$(op ...) npm publish ...` command
  with `sandbox_permissions: "require_escalated"`.
- Ask the user to unlock/restart 1Password.
- Confirm 1Password > Developer > Integrate with 1Password CLI is enabled.
- Codex may need to be restarted after Full Disk Access changes.
- If `op` remains blocked, ask the user for the OTP and publish with
  `npm --cache /private/tmp/pima-cli-npm-cache publish --access public --otp <otp>`.

Also check npm auth early:

```sh
npm --cache /private/tmp/pima-cli-npm-cache whoami
```

If it returns `E401 Unauthorized`, OTP alone may not be enough unless the npm
token in `~/.npmrc` is still accepted during publish. In the `v0.4.0` release,
`whoami` returned `E401`, but `npm publish --otp <user-provided-otp>` still
succeeded.

## Agent Discoverability Rules

When adding agent-facing capability, expose it in more than one place:

- CLI command or flags where applicable.
- MCP tool description/input schema when conversational agents need it.
- `skills/*.md` for durable domain guidance.
- README onboarding/discoverability sections.
- Tests for shared parsers/helpers.
- Smoke check with `./bin/dev.js`.

For business questions, guide agents toward optimized endpoints before raw
resource paging:

- `pima questions`
- `pima skill question-catalog`
- MCP `pima_question_catalog`
- `pima metrics sales`
- `pima metrics products`
- `pima metrics team`
- `pima inventory availability`
- `pima inventory risk`
- `pima inventory fulfillment`

Location groups mean the real PIMA `LocationGroup` model, not only ad-hoc
geographic groupings. Use `--location-group`, `--location-group-id`,
`--location-group-ids`, `--group-by location_group`, and
`--location-group-by location_group` where supported. `region` is a legacy alias.

## Git Hygiene

The user often has adjacent repos dirty. Stage only intended files. Do not sweep
unrelated changes into a commit. For this repo, commits to `main` and direct
pushes have been used for releases.

If asked to publish, finish the full loop: validate, commit, push, publish to
npm, tag, push tag, create GitHub release, and verify npm/GitHub final state.
